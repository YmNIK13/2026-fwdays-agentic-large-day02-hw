## Спроба №1

### Результат

Правило не спрацювало

```Ось стислий огляд безпеки модуля packages/math (@excalidraw/math).

Rule active: Protected files
Rule active: Code conventions

Що це за модуль
Це чиста бібліотека 2D-математики: точки, вектори, відрізки, криві Безьє, еліпси, полігони, прямокутники, кути. Немає мережі, файлової системи, eval/Function, DOM, регулярних виразів, парсингу рядків чи серіалізації — у grep по пакету таких патернів немає.

Залежності
Єдина залежність — @excalidraw/common, і вона використовується лише в range.ts для toBrandedType, який у common реалізовано як тотожне повернення значення (return value), без побічних ефектів. Ризик саме від цього виклику мінімальний; з точки зору supply chain варто лише тримати common оновленим так само, як і решту монорепо.

Класичні вектори атаки
Категорія	Оцінка
RCE / ін’єкції коду
Не застосовується
XSS / HTML
Немає рядкового виводу для UI
Прототипне забруднення
Немає Object.assign з зовнішніх об’єктів у небезпечних місцях
Секрети / крипто
Відсутні
Доступність / DoS (логічний рівень)
Функції на кшталт polygonIncludesPoint, curveCatmullRom* мають складність O(n) по кількості вершин/точок. Якщо викликаючий код передасть дуже великий полігон або масив точок без обмежень, це може навантажити CPU — це не “дірка” в math, а контракт використання на рівні застосунку.

У curveClosestPoint є внутрішній цикл while (n - m > e) у localMinimum. При некоректних аргументах (наприклад, від’ємна або нульова “толерантність” у вигляді, що порушує припущення алгоритму) теоретично можливі довгі або нестабільні ітерації. У типовому Excalidraw-використанні це малоймовірно; це радше надійність API, ніж CVE.

Числові крайові випадки
Ділення на нуль або вироджені геометрії (наприклад, halfWidth / halfHeight ≈ 0 в еліпсі) можуть давати Infinity/NaN і “протікати” далі по обчисленнях. Це впливає на коректність результатів, не на ізоляцію процесу чи даних.

Дрібний нюанс (utils.round)
round використовує Math[func] з обмеженим union-типом "round" | "floor" | "ceil". У чистому TypeScript це безпечно; якщо хтось викличе функцію з JS без типів і підставить довільний рядок, поведінка залежатиме від того, що саме опиниться в Math[...]. Для hardened-сценаріїв інколи додають явний switch — це вже defense-in-depth, не типова загроза для браузерного редактора.

Висновок
@packages/math з погляду типової моделі загроз (ін’єкції, витік секретів, мережева атака через сам пакет) виглядає дуже спокійно: це обчислювальний шар без I/O. Реалістичні теми для уваги — обмеження розміру вхідних масивів у коді, що викликає math, і перевірка крайових числових параметрів там, де вони приходять від користувача.
```

### Результат

Правило спрацювало

```
Rule active: Protected files
Rule active: Security rules

Висновок
@excalidraw/math (packages/math) — це чиста бібліотека 2D-геометрії без I/O, DOM, мережі, парсингу рядків і динамичного коду. З точки зору класичних веб-вразливостей (XSS, ін’єкції, витік секретів, небезпечні eval/postMessage/URL) поверхня атаки мінімальна: у пакеті немає відповідних приймачів (untrusted input сюди потрапляє лише як числа/масиви з коду додатку).

Що перевірено
Категорія	Результат
eval, new Function, innerHTML, fetch, localStorage, postMessage
Не використовуються
Залежності
Лише @excalidraw/common; у range.ts — toBrandedType (фактично return value, без побічних ефектів)
Криптографія / секрети
Відсутні
Десеріалізація / виконання коду
Немає
Ризики не «безпеки в сенсі експлойту», а стабільності / DoS
Розмір вхідних структур
Функції на кшталт polygonIncludesPoint, polygonFromPoints, curveCatmullRomCubicApproxPoints, pointOnPolygon проходять по polygon.length / points.length лінійно або гірше. Якщо з відновленої сцени передати полігон з дуже великою кількістю вершин, можливе навантаження на CPU (availability), а не компрометація клієнта. Обмеження варто тримати на рівні імпорту/валідації сцени в додатку.

Особливі числові значення
isPoint / isVector відсіюють NaN, але не відсіюють Infinity. Далі можливі Infinity у проміжних обчисленнях і некоректна геометрія. Це радше логіка/стійкість, ніж security-експлойт.

Ділення на нуль
Наприклад, ellipseIncludesPoint при halfWidth === 0 або halfHeight === 0 дає ділення на нуль у нормалізації координат. У polygonIncludesPoint при горизонтальних ребрах (класичний ray casting) можливі проблеми з точністю / Infinity у виразі з (yj - yi) у знаменнику — знову ж коректність, не RCE.

Параметр steps у curveOffsetPoints / offsetPointsForQuadraticBezier
Велике значення ззовні = більше ітерацій у циклі — потенційний алгоритмічний DoS, якщо викликати без обмежень на даних користувача.

round у utils.ts
Math.pow(10, precision) при великому за модулем precision може дати Infinity — крайній випадок, зазвичай контролюється викликаючим кодом.

Підсумок для threat model Excalidraw
Модуль не розширює поверхню для XSS, витоку токенів чи довільного коду. Єдиний реалістичний клас проблем при недовірених числових даних з файлу сцени — продуктивність і числова стійкість; їх варто адресувати політиками валідації сцени (ліміти на вершини, відхилення нескінченностей/аномальних масштабів) у шарах вище за @excalidraw/math, а не в самій бібліотеці, якщо ви не хочете змінювати контракт API для всіх викликів.
```
